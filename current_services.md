## DynamoDB Tables

### pa_requests
**Owner:** [Agent Pipeline](agent_pipeline.md)
**Purpose:** Central pipeline state table. Every pipeline step reads and updates this record. Dashboards query it for PA search and visualization.
**Key Schema:** PK: `pa_request_id` (S)
**Billing:** PAY_PER_REQUEST
**Streams:** Enabled (NEW_AND_OLD_IMAGES) -- powers real-time WebSocket updates to dashboards.

### pa_memories
**Owner:** [Memory Feature](memory_feature.md)
**Purpose:** Stores learnings from successful and failed PA applications to improve future submissions.
**Key Schema:**
- PK: `memory_type` (S) -- values: `general`, `provider`, `treatment`, `treatment_provider`
- SK: `memory_id` (S)
**Billing:** PAY_PER_REQUEST
**GSIs:**
- `gsi-provider-created`: PK `provider` (S), SK `created_at` (S), Projection ALL
- `gsi-treatment-created`: PK `treatment` (S), SK `created_at` (S), Projection ALL
- `gsi-provider-treatment-success`: PK `provider_treatment` (S), SK `success_count` (N), Projection ALL
**Embeddings:** Items store an `embedding` attribute (list of float, 1024 dimensions) generated by Bedrock Titan Embeddings v2 for semantic search via agentic RAG.

### pa_scrape_cache
**Owner:** [Search Service](search_service.md)
**Purpose:** TTL-based cache for web scraping results. Prevents redundant scraping on repeated searches.
**Key Schema:** PK: `cache_key` (S)
**Billing:** PAY_PER_REQUEST
**TTL:** Enabled on `ttl` attribute (epoch seconds, auto-expires stale entries)

### pa_patients
**Owner:** [Patient Data](patient_data.md)
**Purpose:** Source-of-truth patient records. Written by Physician Dashboard (patient creation form). Read by Agent Pipeline (Step 0 hydration), Physician Dashboard (patient selector), Pipeline Dashboard (patient search).
**Key Schema:** PK: `patient_id` (S)
**Billing:** PAY_PER_REQUEST
**GSIs:**
- `by_physician`: PK `primary_physician_id` (S), SK `last_name` (S), Projection ALL — patients for a given physician, alphabetical
- `by_name`: PK `last_name` (S), SK `first_name` (S), Projection ALL — find patient by name

### Insurer Portal
**Owner:** [Agent Pipeline](agent_pipeline.md)
**Purpose:** Web-based interface for insurers to review and decide on PA requests, replacing the previous SES email workflow.
**Route:** `/insurer` -- lists pending PA requests for review.
**API:** `POST /api/insurer/pa-requests/{id}/decide` -- submit approval or rejection with optional reasons.
**Flow:** After pipeline Step 6 (submission), status becomes `pending_insurer_review`. Insurer reviews the request and attached PDF via the portal, then approves or rejects. Decisions update `pa_requests` and trigger downstream pipeline logic (memory saving, appeal on rejection).
